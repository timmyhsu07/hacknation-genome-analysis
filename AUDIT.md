# GENOME FIREWALL — Requirements Audit (Phase 1)

> **Update, post-Phase-1:** at the user's direction, two findings below were fixed before Phase 2/3 formally began, ahead of the original phase gating:
> 1. **Cross-cutting finding #1 ("Modules 2 and 3 have never been run together")** is resolved. `module3_decision_report/src/decision_report/real_pipeline.py` now implements Module 3's `FeatureExtractor`/`Predictor` protocols against real Module 1 output and real Module 2 `.joblib` artifacts, proven by `module3_decision_report/tests/test_real_pipeline.py` (5 passing tests, no mocks) and `scripts/demo_real_pipeline.py`. `covered_drugs()` is now derived from the trained artifacts rather than hardcoded, which also resolves the SCOPE-table finding that the covered drug set was declared twice, inconsistently.
> 2. **The `on_target_absent: "susceptible"` footgun** (Module 02 row, "Deterministic target gate") is resolved. `module2_predictor/src/predictor/config.py` now rejects `"susceptible"` at config-validation time; `contracts/model_artifact.schema.json`'s enum was updated to match; a test (`module2_predictor/tests/test_config.py`) covers both the rejection and the two remaining safe values.
>
> Everything else in this document (the Streamlit/Gradio gap, Module 1's missing target-gene capture, the OOD cross-fold evaluation leak, the missing explicit `no_call_rate` field, etc.) is **unchanged and still open** — see the root `README.md`'s "Honest limitations" section for the current, up-to-date summary of what remains.

**Scope of this document:** read-only audit. Nothing in the codebase was changed to produce this report. Every claim below cites a `file:line`; where no citation could be produced, the status is MISSING regardless of what a docstring or README claims.

Status legend:
- **IMPLEMENTED** — the requirement is met, wired into a real control-flow path, and evidence traces the actual execution, not just a comment.
- **PARTIAL** — exists but incomplete, hardcoded where it should be tunable, unenforced where it should be validated, or inconsistent across modules.
- **MISSING** — no citable code implements this.
- **STUBBED-BUT-UNUSED** — the code exists but nothing in the repo calls it on a real path (mocks/tests don't count as "used" for this label).

## Cross-cutting adversarial findings (not explicit checklist rows, but material)

1. **Modules 2 and 3 have never been run together.** No file in the repo imports both `predictor` and `decision_report`. Every Module 3 test, demo, and evaluation exercises `mock_pipeline.py`'s synthetic predictor, never a real `.joblib` artifact. The "three-module pipeline" described in the brief exists as three *independently correct* pieces, not as one integrated system. This single fact downgrades several rows above from IMPLEMENTED to "IMPLEMENTED against mocks only."
2. **Drug/species set declared inconsistently.** Module 2's real config (`ciprofloxacin`, `ampicillin`, `gentamicin`) and Module 3's mock (`Ciprofloxacin`, `Ampicillin`, `Gentamicin`, `Trimethoprim-sulfamethoxazole`, `Colistin`) don't match. There is no single file a reviewer could point to as "the covered set."
3. **`on_target_absent: "susceptible"` is a live footgun.** Nothing rejects it at config-validation time (`module2_predictor/src/predictor/config.py:119-126`); only current data discipline (all three reference drugs use `no_call`) keeps the deployed behavior safe.
4. **OOD reference set leaks other test-fold genomes during cross-validated evaluation.** In `module2_predictor/src/predictor/train.py:151` (`_fold_records`), the out-of-distribution nearest-neighbor check at line 191 (`reference_idx = ood_reference_idx[ood_reference_idx != idx]`) computes distance against `ood_reference_idx`, which is `known_idx` — **every** genome with a known label for that drug, across **all folds**, not just the current fold's fit set (`train.py:380`, passed through at `train.py:398`). Since whole clusters are held out together, a genuinely novel test-fold cluster with more than one member will find its own near-duplicate replicate still sitting in the "reference" pool (only the literal same-row `idx` is excluded), which can make a truly out-of-distribution genome look in-distribution during the **reported OOF metrics** — inflating apparent accuracy/no-call behavior in `metrics_report.json`. This does not affect the final deployed artifact's `train_fingerprints` (`train.py:293`, correctly built from `known_idx` at deploy time, where there is no test set to leak from) — it only affects the honesty of the **evaluation numbers**, which is exactly what the brief cares about. No test in `module2_predictor/tests/` currently catches this.
